# Interface contract: CDS risk dashboard

## Artifact

- Prototype: `cds-risk-dashboard`
- Interface name: static synthetic CDS risk dashboard
- Interface version: default HTML output
- Owner: repository prototype
- Last updated: 2026-07-06

## Safety boundary

This interface is a static synthetic research sketch. It is not
medical advice, not clinical decision support, and not a
validated workflow tool.

## Workflow context

- Intended user: repository reader evaluating workflow design.
- Workflow moment: simulated review point before any real action.
- User action being studied: reading risk, uncertainty,
  limitations, and override context in one surface.
- Action this interface must not request: diagnosis, treatment,
  escalation, discharge, or any patient-specific action.
- Time pressure: illustrative only.
- Interruption level: passive dashboard, not an interruptive
  alert.

## Inputs

- Source artifact: synthetic patient-like records generated by
  `generate_dashboard.py`.
- Required fields: synthetic identifier, risk display,
  confidence band, explanation factors, limitations, and audit
  event fields.
- Refresh cadence: static regeneration by script.
- Missing input behavior: current sketch uses complete example
  rows only.
- Invalid input behavior: not implemented.

## Outputs

- Primary display: synthetic risk category and score.
- Uncertainty display: confidence wording for the score.
- Explanation display: contextual factors with caveats.
- Limitation display: visible limitations beside the estimate.
- Audit event: example override and review log row.

## Human control

- Override path: represented through example audit fields.
- Required reason capture: override reason in audit output.
- Accountability boundary: user remains responsible in the
  design sketch, but no real decision is supported.
- Escalation path: governance review only, not clinical
  escalation.

## Failure modes

- Known misuse: reading the risk display as real clinical advice.
- Alert fatigue risk: low in current passive sketch, but future
  variants should compare passive, inline, and interruptive
  forms.
- Over-trust risk: explanations may look more certain than the
  synthetic data allows.
- Under-trust risk: uncertainty wording may make every output
  look equally unusable.
- Fallback behavior: do not use the interface for real decisions.

## Evaluation

- Usability check: not yet performed.
- Workflow burden check: not yet performed.
- Audit completeness check: example audit log exists.
- Local validation precondition: real-data validation,
  calibration, subgroup review, workflow study, and governance
  approval would all be required before any deployment
  conversation.
